# GTD × Apple Reminders 技术实施清单 v1.0

> 基于《重构方案1.0》与《requirements v1.1》输出的实施文档。目标是把方案和需求转成可执行的开发任务清单。

---

## 一、文档目标

这份文档回答的是：

1. 具体先改哪些文件
2. 每一步做什么
3. 每一步做完怎么验证
4. 哪些可以并行，哪些必须串行
5. 做完后系统应达到什么状态

这是开发排期与执行的直接依据。

---

## 二、总体实施策略

本轮采用：**收敛式重构，分 4 个阶段推进**。

阶段顺序：
1. 状态语义收口
2. 视图与主清单规则收口
3. GTD -> Apple 单向同步收口
4. Apple completed -> GTD done 回写收口

原则：
- 尽量保留现有主库与 CLI
- 尽量不推倒重写
- 每一步都可单独验证
- 每一步都可回退

---

## 三、阶段拆解

# Phase 1：状态语义收口

## 目标
统一 `open / done / deleted` 语义，彻底区分 done 和 delete。

## 需要处理的文件

### 重点文件
- `data/tasks.json`
- `scripts/task_cli.py`
- `scripts/render_views.py`
- `README.md`

## 任务清单

### 任务 1.1：统一任务状态语义

**目标**：明确代码中的状态行为。

**要求**：
- `done`：表示完成，保留记录
- `deleted_at != null`：表示真删除
- 主逻辑中不再把“完成”当删除使用

**建议动作**：
- 检查 `task_cli.py` 中 `done / update / delete / reopen` 相关逻辑
- 若缺少真正删除命令，补一个明确 delete 语义入口
- delete 应只写 `deleted_at`，而不是伪装成 done

**验收**：
- done 后任务不在主清单，但在 done.md 可见
- delete 后任务主清单不可见，done.md 也不可见

---

### 任务 1.2：统一渲染过滤规则

**目标**：所有视图遵守同一显示规则。

**要求**：
- 主清单只显示 `status=open && deleted_at is null`
- done 只进入完成区
- deleted 不进入常规视图

**建议动作**：
- 在 `render_views.py` 中统一封装过滤函数
- 不要在不同视图里各写各的判断

**验收**：
- today / inbox / matrix / weekly open 部分不出现 deleted 或 done 任务
- done.md 只显示 done/cancelled/archived（如保留）
- deleted 项不出现在常规视图

---

### 任务 1.3：更新文档定义

**目标**：文档与代码行为一致。

**建议动作**：
- 更新 `README.md`
- 补充 done / delete 的定义
- 补充主清单显示规则

**验收**：
- README 中产品语义与代码行为一致

---

# Phase 2：视图与使用入口收口

## 目标
确保主库是唯一入口，视图只是视图，日常操作尽量走受控入口。

## 需要处理的文件
- `scripts/task_cli.py`
- `scripts/nlp_capture.py`
- `scripts/render_views.py`
- `README.md`

## 任务清单

### 任务 2.1：确认 CLI 为主要受控入口

**目标**：避免长期手改视图和主库。

**建议动作**：
- 检查 CLI 是否覆盖 add / update / done / reopen / delete
- 若 delete 缺失，则补齐
- 若 list 过滤能力不够，可补 `--deleted` / `--status done` 等

**验收**：
- 常见操作都可通过 CLI 完成
- 不需要手工改 `tasks.json` 或 markdown 视图

---

### 任务 2.2：检查 NLP 入口是否遵守新语义

**目标**：NLP apply 后，不破坏状态语义。

**建议动作**：
- 检查 `nlp_capture.py` 是否仍假设某些旧语义
- 确保新增任务默认是 open
- 不让 NLP 直接混入 delete 语义

**验收**：
- `preview/apply` 结果符合新规则

---

# Phase 3：GTD -> Apple 单向同步收口

## 目标
把单向同步做成真正稳定可用的能力。

## 需要处理的文件
- `scripts/sync_apple_reminders.py`
- `scripts/export_apple_reminders_sync.py`
- `scripts/apple_reminders_sync_lib.py`
- `sync_apple_reminders_mac.applescript`
- `mac/run_apple_reminders_sync.sh`
- `sync/apple-reminders-sync-state.json`

## 任务清单

### 任务 3.1：收敛同步职责

**目标**：同步脚本只做“必要的事”。

**建议动作**：
- 明确 `sync_apple_reminders.py` 是编排入口还是核心逻辑入口
- 保留一个统一入口，但内部职责拆清楚：
  - export/push
  - status
  - （未来）completed pull

**验收**：
- 同步脚本职责清晰
- 调用路径简单可理解

---

### 任务 3.2：收敛状态文件字段

**目标**：状态文件只保留必要信息。

**保留建议**：
- gtd_id
- last exported signature
- last target list
- last pushed at
- last push status
- last error

**建议动作**：
- 清理明显“为未来复杂同步预埋但当前不用”的字段
- 保证状态文件可读可审计

**验收**：
- 状态文件能看懂
- 可以快速定位同步失败原因

---

### 任务 3.3：验证去重机制

**目标**：避免重复创建 reminders。

**建议动作**：
- 统一 `[GTD_ID]` 作为稳定锚点
- 导出和 Mac 端写入都依赖稳定 ID
- 不允许靠标题模糊匹配创建新 reminder

**验收**：
- 重复执行同步不会生成多个 reminder

---

### 任务 3.4：稳住 Mac 执行链路

**目标**：Mac 桥接真正可长期跑。

**建议动作**：
- 检查 `mac/run_apple_reminders_sync.sh`
- 确保执行顺序明确：
  1. git pull
  2. 消费 export
  3. 写 Apple Reminders
- pull 失败时，保守降级但不中断本地已有 export 的消费

**验收**：
- Mac 本地定时执行稳定
- 工作区脏时不会误破坏仓库
- pull / push / applescript 失败时日志可查

---

# Phase 4：Apple completed -> GTD done 回写收口

## 目标
只增加一个最小但高价值的回写能力：completed -> done。

## 需要处理的文件
- `mac/export_completed_reminders_phase1.applescript`
- `scripts/consume_apple_reminders_completed.py`（或重命名为更统一名字）
- `sync/apple-reminders-completed-events.json`
- `sync/apple-reminders-completed-applied.json`
- `scripts/render_views.py`

## 任务清单

### 任务 4.1：统一 completed 事件格式

**目标**：事件文件结构固定，便于幂等消费。

**建议字段**：
- event_id
- event_type=completed
- gtd_id
- completed_at
- title
- source

**验收**：
- 事件文件结构固定
- 没有 gtd_id 的事件直接跳过

---

### 任务 4.2：完成事件消费器幂等化

**目标**：重复消费不会污染主库。

**建议动作**：
- 使用 `applied_event_ids` 去重
- 若任务已 done，则记为已消费并跳过
- 消费成功后刷新视图

**验收**：
- 相同 completed 事件重复导入不会重复改写
- 已 done 任务不会报错

---

### 任务 4.3：回写链路与主同步链路解耦

**目标**：completed 回写不污染单向同步主链。

**建议动作**：
- completed 回写保持独立脚本
- 不要把它直接混进 export 自动提交流程
- 如需提交 Git，走独立 commit 语义

**验收**：
- completed 回写出问题时，不影响 GTD -> Apple 主链

---

## 四、推荐代码调整顺序

### 第一批（今天就能做）
1. 统一 done / delete 语义
2. 修正 render_views 过滤逻辑
3. 补 delete CLI 语义（如果缺失）
4. 更新 README

### 第二批
5. 收敛 sync state 字段
6. 稳住 GTD -> Apple 单向同步链路
7. 补 `[GTD_ID]` 稳定锚点校验

### 第三批
8. 完成 completed 事件格式定稿
9. 完成 completed 回写脚本幂等化
10. 完成 Mac 回写链路验证

---

## 五、建议测试清单

## 5.1 状态语义测试
- 新增任务 -> open
- done 任务 -> 主清单消失，done.md 出现
- delete 任务 -> 主清单消失，done.md 不出现
- reopen done 任务 -> 回到 open

## 5.2 单向同步测试
- 新增任务 -> Apple 新增 reminder
- 改标题 -> Apple 标题更新
- done -> Apple completed
- 重复执行同步 -> 不重复创建

## 5.3 回写测试
- Apple 勾选完成 -> GTD done
- 相同 completed 事件重复导入 -> 无副作用
- Apple 修改标题 -> GTD 不回写
- Apple 删除 -> GTD 不删除

---

## 六、并行与串行建议

## 必须串行
- Phase 1（语义收口）必须先做
- Phase 3 前必须完成视图规则收口
- Phase 4 必须在单向同步稳定后再做

## 可并行
- README 更新 与 render_views 重构可并行
- Mac completed 导出脚本 与 Linux 消费脚本可并行
- 同步状态文件收敛 与测试用例整理可并行

---

## 七、回退策略

每个阶段都要能单独回退：

### Phase 1 回退
- 保留旧字段兼容
- 先不删旧逻辑，只修改展示规则

### Phase 3 回退
- Mac 同步失败时仍允许退回手动同步
- launchd 可停用
- git pull 可关闭

### Phase 4 回退
- completed 回写脚本单独关闭
- 不影响主库写入和单向同步

---

## 八、实施完成后的目标状态

完成后，系统应达到：

1. `tasks.json` 是唯一事实源
2. done 与 delete 语义彻底分开
3. 主清单只显示 open
4. GTD -> Apple 单向同步稳定可用
5. Apple -> GTD 只支持 completed -> done
6. 系统不依赖独立服务端
7. 日志与状态文件足够排障，但不过度复杂

---

## 九、项目管理建议

建议把本轮实施拆成 3 个开发包：

### 开发包 A：状态语义与视图收口
- done / delete
- render_views
- README

### 开发包 B：单向同步收口
- sync state
- export/push
- Mac run script

### 开发包 C：completed 回写收口
- completed events
- consume script
- 幂等与验证

这样每一包都可以独立验收。

---

## 十、最终建议

本轮不追求“功能更多”，而追求：

> **边界更清楚，链路更短，运行更稳。**

正确实施顺序不是“继续扩功能”，而是：

1. 先收口语义
2. 再收口单向同步
3. 最后只加 completed 回写

这才是最符合当前项目阶段的技术路径。
