# GTD Tasks v0.2.2 需求文档

## 1. 背景

`gtd-tasks` 在 v0.2.1 已完成一次关键收口：

- `data/tasks.json` 成为主账号 GTD 的唯一事实源
- `today.md / inbox.md / matrix/*` 退化为渲染视图，而非手工维护的数据源
- wife 账号的数据层同步完成初始化
- 时间口径统一为 `Asia/Shanghai`

这意味着项目已经从“markdown 管理待办”升级到“结构化数据管理待办”。

但 v0.2.1 仍然偏底层能力落地，距离“真正顺手可长期使用的 GTD 系统”还差一层：

1. 缺少完整的任务操作闭环
2. 缺少自然语言到结构化任务的入口
3. 缺少更适合日常回顾和决策的派生视图
4. 文档和版本说明尚未围绕新能力更新

因此，v0.2.2 的目标不是继续扩张系统边界，而是补齐“可用性层”。

---

## 2. 版本目标

v0.2.2 的目标可以概括为一句话：

**让 `gtd-tasks` 从“已经有主库”升级为“可以顺手维护、顺手录入、顺手查看”的日常系统。**

本版本重点聚焦三件事：

1. 任务操作闭环
2. 自然语言录入入口
3. 派生视图与回顾能力

其中优先级顺序为：

- P0：任务操作闭环
- P1：自然语言录入
- P1：派生视图增强
- P2：文档与版本说明更新

---

## 3. 本版本范围

### 3.1 In Scope

本版本纳入范围的内容：

#### A. CLI 操作闭环
- 补齐主账号任务的增删改查体验
- 支持按条件筛选任务
- 支持移动 bucket
- 支持修改 quadrant
- 支持更新 note
- 支持维护 tags
- 支持任务完成、取消、重开
- 支持批量操作的最小闭环

#### B. 自然语言录入入口
- 支持从一句自然语言生成结构化任务
- 至少支持提取：
  - title
  - bucket（today/tomorrow/future）
  - tags
  - note
  - quadrant（可默认推断或默认值）
- 支持“预览后确认写入”模式
- 支持“直接写入”模式

#### C. 视图增强
- 在现有 `today / inbox / matrix` 之外，增加更适合复盘和管理的视图
- 优先考虑：
  - `done.md`：已完成视图
  - `weekly-review.md`：周回顾视图
  - `overdue.md` 或等待后续版本的数据字段支持后再定

#### D. 文档更新
- README 同步更新为 v0.2.2 口径
- 增补版本说明文档
- CHANGELOG 更新

---

### 3.2 Out of Scope

本版本明确不做的内容：

1. iOS 双向同步
2. 多端冲突处理
3. 飞书原生任务系统深度打通
4. 复杂权限模型
5. 多用户统一主库合并
6. 自动化 AI 决策分派
7. 复杂项目/子任务系统

说明：
这些内容要么复杂度明显更高，要么需要先等 v0.2.2 把单机主库和使用体验打稳后再推进。

---

## 4. 现状问题

### 4.1 数据层已经落地，但操作层还偏原始

虽然已有 `task_cli.py`，但在 v0.2.1 的定义下，CLI 仍更像一个最低可用脚本，而不是一个顺手的任务管理入口。

当前问题包括：
- 查询能力有限
- 更新维度有限
- 标签能力弱
- 缺少批量操作
- 缺少 reopen 等常见动作

### 4.2 用户输入仍然过于“命令式”

当前系统更适合：

```bash
python3 scripts/task_cli.py add "给张闯回信" --bucket future
```

但真实使用场景更可能是：

- 明天提醒我给张闯回信
- 把海南公司主体先放未来
- 这件事我自己做，打个 ME 标签
- 这个先完成

也就是说，系统目前适合脚本，不够适合真实对话式使用。

### 4.3 视图存在，但管理视角还不够完整

当前已有：
- today
- inbox
- matrix

但缺少：
- 已完成集中查看
- 周回顾生成
- 更适合“看最近推进结果”的视图

## 5. 用户故事

### 用户故事 1：快速录入
作为用户，我希望直接说一句自然语言，就能形成结构化任务，而不是每次手写 CLI 参数。

### 用户故事 2：快速调整
作为用户，我希望能很方便地把任务从 `today` 改到 `future`，或者补一个标签、备注，而不是手动改 JSON。

### 用户故事 3：快速回看
作为用户，我希望能快速看到“今天要做什么”“最近完成了什么”“本周推进了什么”，而不是只看到一个扁平任务列表。

### 用户故事 4：AI 可代理执行
作为 AI 助手，我希望任务系统有稳定的数据结构和操作接口，这样我才能可靠地帮用户记任务、改任务、出视图、做提醒。

## 6. 功能需求

## 6.1 模块 A：任务操作闭环

### A1. list 增强
系统应支持按以下条件筛选任务：
- id
- status
- bucket
- quadrant
- tag
- text（title/note 模糊匹配）

系统应支持：
- 简洁列表模式
- verbose 模式
- limit 限制返回数量

### A2. update 增强
系统应支持对单个任务更新以下字段：
- title
- bucket
- quadrant
- note
- status
- tags

### A3. done / reopen
系统应支持：
- 标记完成
- 从完成状态恢复为 open
- reopen 时允许指定 bucket

### A4. batch move
系统应支持按筛选条件批量移动任务 bucket，例如：
- 把所有 `today` 中带 `ME` 标签的任务移到 `future`
- 把文本包含“规划”的任务从 `tomorrow` 移到 `today`

### A5. batch tag
系统应支持按筛选条件批量执行：
- 添加 tag
- 删除 tag
- 重置 tag

### A6. 数据字段维护
系统在变更任务时应自动维护：
- updated_at
- completed_at
- sync_version

---

## 6.2 模块 B：自然语言录入

### B1. 输入形式
系统应提供自然语言录入脚本或入口，例如：

```bash
python3 scripts/nlp_capture.py "明天提醒我给张闯回信"
```

### B2. 最低识别范围
系统至少应能识别：
- 标题 title
- 时间归属 bucket：today / tomorrow / future
- tags（如 `#ME`、`我来处理`）
- note（如“下周实施”“等确认”）

### B3. 输出模式
系统至少应支持两种模式：

1. preview
   - 仅展示解析结果
   - 不写入主库

2. apply
   - 展示解析结果
   - 写入主库
   - 自动触发 render

### B4. 容错策略
若无法可靠识别时间归属：
- 默认进入 `future`
- 或允许用户显式指定默认 bucket

若无法识别 quadrant：
- 默认 `q2`

### B5. 非目标
本版本不追求复杂 NLP 能力，不要求：
- 多轮对话推理
- 复杂日期解析库接入
- 高级项目识别
- 子任务自动拆分

本版本的目标是：**够用、稳定、可控。**

---

## 6.3 模块 C：视图增强

### C1. done 视图
系统应新增已完成任务视图，例如 `done.md`，用于集中查看：
- 最近完成任务
- 已取消/已归档任务

### C2. weekly review 视图
系统应新增周回顾视图，例如 `weekly/review-latest.md`，至少包含：
- 本周新增任务数
- 本周完成任务数
- 当前未完成任务概览
- 按 bucket 分类的待办概览

### C3. 与现有 render 机制兼容
新视图应由渲染脚本自动生成，不能成为新的手工事实源。

## 7. 非功能需求

### 7.1 稳定性
- 不能破坏现有 `data/tasks.json` 结构兼容性
- 旧数据应可无迁移或轻迁移继续使用

### 7.2 可维护性
- 尽量保持脚本简单直接
- 不引入明显过重的依赖
- 尽量使用标准库实现

### 7.3 可解释性
- 自然语言解析结果必须可预览
- 不能让用户在不知情的情况下误写数据

### 7.4 一致性
- 所有时间语义继续统一按 `Asia/Shanghai`
- 渲染视图继续遵守“展示层不是事实源”的原则

---

## 8. 建议实现顺序

### Phase 1：CLI 闭环
目标：先把结构化任务维护能力补齐。

交付物：
- 增强版 `task_cli.py`
- 对应 README 用法更新

### Phase 2：自然语言录入
目标：让对话式录入真正可用。

交付物：
- `scripts/nlp_capture.py`
- preview / apply 模式
- 基础规则解析

### Phase 3：视图增强
目标：让系统更适合回顾和管理。

交付物：
- `done.md`
- `weekly/review-latest.md`
- `render_views.py` 更新

### Phase 4：文档收口
目标：把版本信息和使用方式同步清楚。

交付物：
- `README.md` 更新
- `CHANGELOG.md` 更新
- `release-v0.2.2.md`

---

## 9. 验收标准

当满足以下条件时，可认为 v0.2.2 达到预期：

### CLI 侧
- 可以稳定新增、更新、完成、重开任务
- 可以按 bucket / tag / text 等维度查任务
- 可以做批量 move / tag

### NLP 侧
- 至少 10 条常见中文输入能正确解析 bucket
- 支持 preview
- 支持 apply 后落库并自动 render

### 视图侧
- 新增 done / weekly review 视图
- 视图生成不破坏现有 today/inbox/matrix

### 文档侧
- README、CHANGELOG、release 文档同步完成

## 11. 一句话总结

v0.2.1 解决的是“数据有没有主库”的问题。  
v0.2.2 要解决的是“这个主库是不是已经真的能被顺手使用”的问题。
