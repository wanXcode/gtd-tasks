# GTD × Apple Reminders 重构方案 1.0

> 目标：基于现有 GTD 项目，做一次**收敛式重构**，把系统收口到“简洁、稳定、不复杂、可长期使用”的状态。

---

## 一、结论

本次建议采用：**重构，不重写**。

原因：
- 当前 `data/tasks.json` 主库思路是对的
- CLI / 视图 / GTD 分类体系已经成型
- Mac 本地执行 AppleScript 的方向是对的
- Git + Mac pull 的保守同步链路是可用资产
- 已有日志、状态文件、导出脚本，可以继续利用

因此，不建议推倒重来；建议做一次**产品先收口、技术后收口**的渐进式重构。

---

## 二、重构目标

本次重构追求 4 个核心目标：

1. **单主库**：`data/tasks.json` 永远是唯一事实源
2. **低复杂度**：不引入独立服务端，不引入复杂双向同步中心
3. **稳定同步**：GTD -> Apple Reminders 自动同步要可靠可用
4. **最小回写**：Apple -> GTD 只支持 `completed -> done`

一句话总结：

> **GTD 是唯一主库；Mac 本地桥接 Apple Reminders；默认单向同步，唯一允许的回写是 completed；不做独立服务端，不做复杂全字段双向同步。**

---

## 三、当前方案的问题

当前实现并不是完全错误，但存在以下问题：

### 1. 同步链路偏绕
当前链路大致是：
- GTD 变更
- 导出 payload
- Git commit / push
- Mac pull
- AppleScript 执行
- 未来还想双向回写

问题：链路过长，排障成本偏高。

### 2. Git 被当成“事件总线”使用
Git 可以继续保留，但不适合承载越来越复杂的 `events / applied / export / import / reconcile` 逻辑。

### 3. 双向同步边界没有收死
如果继续放开：标题、备注、分类、删除都双向同步，复杂度会迅速爆炸。

### 4. 完成 / 删除语义未彻底分离
当前系统已经有 `done` 和 `deleted_at` 字段基础，但产品语义尚未完全统一，这会直接影响同步策略。

---

## 四、最佳产品方案（重构后的目标形态）

### 4.1 核心原则

#### 原则 1：单主库
- `data/tasks.json` 是唯一事实源
- Apple Reminders 不是第二主库，只是执行 / 展示端

#### 原则 2：Mac 是桥接器，不是服务端
- 不做独立后端服务
- 真正操作 Apple Reminders 的地方仍然是 Mac 本机

#### 原则 3：默认单向同步
- 主通道永远是：`GTD -> Apple Reminders`

#### 原则 4：只允许最小回写
- Apple -> GTD 只支持：`completed -> done`
- 不回写标题、备注、分类、象限、标签

#### 原则 5：删除高风险，默认不双向
- GTD 删除不直接要求 Apple 删除
- Apple 删除不回写 GTD 删除
- 删除动作以保守策略处理

---

## 五、重构后的产品语义

### 5.1 任务状态语义

建议把任务状态统一成以下 3 类语义：

#### A. open
- 正常待办
- 显示在主清单
- 参与 GTD -> Apple 同步

#### B. done
- 已完成
- 不显示在日常待办主清单
- 显示在 `done.md` / `weekly/review-latest.md`
- Apple 侧可以同步为 completed

#### C. deleted
- 真删除
- 不显示在主清单
- 不显示在 done/review
- 不应当作为“完成”的替代方案

技术上建议继续使用：
- `status = open / done / cancelled / archived`（如需兼容现有设计）
- `deleted_at != null` 表示硬删除

但产品语义要明确：

> **done 是“隐藏到完成区”，delete 才是真删。**

## 六、重构后的最佳技术架构

## 6.1 总体架构

```text
GTD 主库 (tasks.json)
        |
        | 结构化写入（CLI / NLP / Agent）
        v
   渲染视图（today / inbox / done / weekly）
        |
        | Git push
        v
   Mac 本地仓库 git pull
        |
        +--> push_to_apple
        |
        +--> pull_completed_from_apple
                |
                v
         apply_completed_to_gtd
                |
                v
             Git push 回主库
```

---

## 6.2 模块职责

### 模块 A：GTD 主库层
保留现有：
- `data/tasks.json`
- `scripts/task_cli.py`
- `scripts/render_views.py`
- `scripts/nlp_capture.py`

职责：
- 结构化维护任务
- 渲染 GTD 视图
- 永远作为唯一事实源

### 模块 B：Mac 桥接层
Mac 本地新增 / 收敛为一个桥接执行器，职责只有两个：

#### 1) Push：GTD -> Apple
把 GTD 的 open / done 状态同步到 Apple Reminders。

#### 2) Pull：Apple -> GTD（仅 completed）
只扫描 Apple 侧已完成事项，识别 `[GTD_ID]` 后回写为 done。

### 模块 C：Git 同步层
保留 Git，但降级其职责：
- 只负责同步主库与必要状态文件
- 不承担复杂事件总线职责

---

## 七、同步边界（必须严格收口）

## 7.1 GTD -> Apple Reminders 支持范围

支持：
- 新建任务
- 标题更新
- 完成状态同步
- reopen / 打开状态同步
- 按 category 映射到不同 Apple List

可选支持：
- note 同步
- due date 同步

不建议优先支持：
- 复杂字段映射
- 高复杂度差异对比

## 7.2 Apple -> GTD 支持范围

第一版只支持：
- `completed -> done`

明确不做：
- Apple 改标题回写
- Apple 改备注回写
- Apple 改分类 / bucket / quadrant 回写
- Apple 删除回写

原因：
- completed 是高价值、低歧义、低风险动作
- 其他字段回写会显著提升复杂度和冲突概率

---

## 八、为什么不做服务端

当前场景下，不建议做独立服务端，原因如下：

### 1. Apple Reminders 的真实执行端仍然是 Mac
即使有服务端，最后还是要靠 Mac 本机 AppleScript 去执行。

### 2. 服务端会平白增加复杂度
如果做服务端，就会额外引入：
- agent 心跳
- 任务队列
- 认证
- 调度
- 错误恢复
- 状态中心

这对于当前项目目标来说，明显过重。

### 3. 与“简洁、稳定、不复杂”的目标冲突
本项目最适合的是：
- 无服务端
- 单主库
- Mac 本地桥接
- 定时 git pull + 执行

结论：

> **Apple Reminders 这种场景，Mac 本地桥接比独立服务端更适合。**

## 九、重构策略：保留什么、删除什么、重构什么

## 9.1 保留项

以下资产建议保留：
- `data/tasks.json` 主库
- `scripts/task_cli.py`
- `scripts/render_views.py`
- GTD 五分类体系
- `done.md / weekly/review-latest.md`
- Mac 本地执行 AppleScript 的方向
- Git + Mac pull 的保守架构

## 9.2 删除 / 冻结项

以下方向建议停止扩张或冻结：
- 复杂全字段双向同步
- 伪服务端化设计
- 把 Git 当复杂事件总线
- 一次性做完整 reconcile / conflict engine
- Apple 删除回写 GTD

## 9.3 重构项

重点重构以下部分：

### A. 任务语义层
- 明确 done vs delete
- 主清单只显示 open
- done 进入完成区
- delete 真删除

### B. 同步模块层
把同步逻辑收口为 3 个清晰模块：

1. `push_to_apple`
2. `pull_completed_from_apple`
3. `apply_completed_to_gtd`

### C. Mac 执行链路
Mac 端形成稳定的周期执行器：
1. git pull
2. push_to_apple
3. export completed events
4. apply completed back to GTD
5. 如有必要 git push

## 十一、分阶段实施计划

## Phase 1：产品规则收口

目标：先把规则定死，不急着大改代码。

交付物：
1. 明确状态语义：open / done / deleted
2. 明确显示规则：主清单只显示 open
3. 明确同步边界：Apple 只回写 completed
4. 明确删除策略：Apple 删除不回写，GTD 删除不自动删 Apple

## Phase 2：单向同步收敛

目标：把 `GTD -> Apple` 做稳。

交付物：
1. Mac 定时 pull 最新主库
2. 稳定执行 AppleScript
3. 记录成功 / 失败日志
4. 避免重复创建 reminder
5. 任务完成后可正确同步为 completed

## Phase 3：completed 回写收敛

目标：只增加一个最小回写能力。

交付物：
1. Mac 导出 completed events
2. Linux 侧幂等消费
3. 对应 GTD 任务标记 done
4. 自动刷新视图
5. 视情况 git push 回主库

## Phase 4：运行观察

目标：稳定运行 2~4 周。

关注项：
- 是否有重复同步
- 是否有漏同步
- 是否有 completed 误回写
- Git pull / push 是否稳定
- Mac 本地工作区是否经常脏

只有在这一阶段稳定后，才考虑是否值得进一步开放更多回写字段。

---

## 十二、是否要做完整双向同步？

当前建议：**不要。**

原因：
- 产品价值最高的是“手机上能看到 + 手机上勾掉后主库也完成”
- 完整双向同步会带来大量复杂度
- 对当前项目目标来说，收益不成比例

建议策略：

> **先把“看得到 + 勾得掉 + 主库能回收完成状态”做到极稳，再谈别的。**

---

## 十三、风险与规避策略

### 风险 1：重复创建 Reminder
规避：
- 使用稳定 `[GTD_ID]` 锚点
- 状态文件中记录同步指纹与目标列表

### 风险 2：完成 / 删除混淆
规避：
- 先统一产品语义
- done 与 delete 严格分开

### 风险 3：Git 造成脏工作区
规避：
- Mac 本地只允许少量同步文件变动
- pull 前自动 restore 可重建运行产物

### 风险 4：Apple 端误操作引发主库污染
规避：
- Apple -> GTD 只允许 completed 回写
- 不开放标题 / 备注 / 分类回写

---

## 十四、最终建议

本项目的正确方向不是“继续做大而全双向同步”，而是：

### 最优收敛方案
- 不做独立服务端
- 不做复杂双向字段同步
- 保留单主库
- 保留 Mac 本地桥接
- GTD -> Apple 为主
- Apple -> GTD 只回写 completed
- done 与 delete 语义彻底分离

一句话落地定义：

> **这是一个以 GTD 主库为核心、由 Mac 本地桥接 Apple Reminders 的轻量同步系统，而不是一个全功能双向任务平台。**
