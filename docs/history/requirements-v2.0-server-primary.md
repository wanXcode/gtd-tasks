# GTD 服务端主库方案需求文档 v2.0

> 方案定位：采用 **A 方案** —— 服务端作为唯一事实源（Source of Truth），本地 `tasks.json` 不再承担主主库角色，而是降级为缓存 / 导出 / 兼容层。目标是替换当前 GitHub 中转同步链路，建立适合高频任务录入、低心智负担、可长期扩展的 GTD 系统。

---

## 一、文档目标

本需求文档用于明确以下问题：

1. 本轮重构的产品目标是什么
2. 为什么要从 GitHub 中转切换到服务端主库
3. 哪些能力属于本轮范围，哪些明确不做
4. 用户最终获得的体验应该是什么样
5. 什么状态算“可以上线使用”

本文件是后续技术方案、开发实现、测试验收的直接依据。

---

## 二、背景与问题定义

当前系统已经具备：

- GTD 数据结构与任务模型
- 本地任务写入能力（CLI / NLP）
- 视图渲染能力（today / inbox / done / weekly）
- Mac 侧 Apple Reminders 桥接能力
- 基于 Git/GitHub 的跨端同步链路

但当前方案存在明显问题：

### 2.1 用户体验问题
- 每次任务录入/修改后，还需要额外提醒 AI 提交到 GitHub
- 用户需要等待 `commit / push / pull / sync` 这条长链路完成
- GitHub 的技术语义暴露给最终使用者，体验不自然

### 2.2 系统复杂度问题
- Git 适合代码版本管理，不适合高频、小颗粒度任务同步
- 多端同时变更时容易产生 rebase / merge conflict
- 冲突处理成本高，且容易污染任务主库

### 2.3 架构方向问题
- 任务系统本质上需要的是“数据服务 + 增量同步”
- 当前方案用“文件仓库同步”代替“任务服务同步”，方向不优

结论：

> 当前系统的核心问题，不在 GTD 业务模型，而在同步架构。

因此，本轮不推倒重写 GTD 业务层，而是：

> **将系统升级为“服务端主库 + Mac 增量同步”的任务平台。**

---

## 三、本轮版本目标

本轮版本目标不是做一个复杂通用任务平台，而是把现有 GTD 系统升级为：

> **以服务端任务库为唯一事实源、由 AI 负责写入、由 Mac 负责同步到 Apple Reminders 的轻量任务系统。**

目标分为 4 个：

### 目标 1：建立唯一主库
- 服务端数据库成为任务的唯一事实源
- 所有任务新增、修改、完成、删除操作都以服务端为准
- 本地 `tasks.json` 不再作为唯一事实源

### 目标 2：降低用户操作心智负担
- 用户与 AI 对话后，任务应直接进入系统
- 用户不需要再关心 GitHub、commit、push、pull
- 用户不应被迫理解中间同步过程

### 目标 3：实现稳定的 Mac 端同步
- Mac 可定时从服务端拉取任务变更
- 将变更同步到 Apple Reminders
- 支持最小必要的 Apple -> 服务端回写

### 目标 4：保留现有系统资产
- 复用现有任务模型
- 复用现有视图渲染规则
- 复用现有 Apple 映射逻辑与经验
- 控制重构成本，避免从零重写

---

## 四、产品原则

### 4.1 单主库原则
服务端数据库是唯一事实源。

这意味着：
- 所有任务真实状态以服务端为准
- 本地 JSON / Markdown 视图都不是事实源
- Apple Reminders 不是主库，只是桥接展示/提醒层

### 4.2 AI 直写原则
用户与 AI 的交互，直接转化为服务端任务写入。

这意味着：
- “记个任务”应直接落到服务端
- 不再要求先写本地、再 GitHub 中转
- 用户不需要显式触发“提交”动作

### 4.3 Mac 本地桥接原则
Apple Reminders 的实际执行端仍然是 Mac 本机。

这意味着：
- 不要求服务端直接控制 Apple
- 仍由 Mac 同步代理负责 AppleScript / Reminders 集成
- Mac 作为服务端与 Apple 之间的桥

### 4.4 保守同步原则
当同步异常时：
- 优先不误删
- 优先不损坏主库
- 优先保留恢复空间
- 优先可观察、可补偿

### 4.5 先收口、后扩张原则
本轮只解决“主链路稳定可用”，不追求过度平台化。

## 五、用户体验目标

用户理想体验应是：

### 场景 1：新增任务
用户说：
> 明天下午 3 点开员工大会

系统应自动完成：
1. AI 解析任务语义
2. 任务直接写入服务端主库
3. 本地视图刷新（若需要）
4. Mac 在同步周期内获取增量
5. Apple Reminders 自动出现对应任务

用户不需要：
- 提醒提交到 GitHub
- 等待手工 push/pull
- 处理同步细节

### 场景 2：修改任务
用户说：
> 不是明天，是今天

系统应自动完成：
1. 找到目标任务
2. 更新服务端字段
3. 同步任务变更到 Mac
4. Apple 端对应任务被更新

### 场景 3：完成任务
用户可在以下任一位置完成任务：
- 对 AI 说“完成这个任务”
- 在 Apple Reminders 中勾选完成

系统应保证：
- 服务端最终状态一致
- 完成记录保留
- 主清单不再展示该任务

## 六、范围定义

## 6.1 本轮必须做

### 任务主库服务端化
- 提供服务端任务存储
- 提供任务的新增/查询/更新/删除接口
- 提供变更日志/增量同步能力

### AI 写入服务端
- AI 对 GTD 的新增、修改、完成、删除能力应切换到服务端
- 至少保证关键常用操作可用

### Mac 增量同步
- Mac 能按固定周期（建议 1 分钟）拉取服务端增量
- 将变更同步到 Apple Reminders

### Apple 最小回写
- 支持 `completed -> done`
- 支持必要的同步游标与映射状态管理

### 本地视图兼容
- 保留 `today.md / inbox.md / done.md / weekly/review-latest.md`
- 这些视图可由服务端数据导出/渲染生成

---

## 6.2 本轮可选做

### due date / remind time 同步
- 若现有模型支持，可纳入
- 若增加过多复杂度，可延后

### Web 管理页
- 非必须
- 后续可补

### 推送式同步（Webhook / SSE / WebSocket）
- 本轮不要求
- 先用轮询实现可靠闭环

---

## 6.3 本轮明确不做

### 不做完整多端协同平台
本轮不目标化实现：
- iPhone 原生客户端
- Web 全功能管理台
- 多用户团队权限系统

### 不做复杂双向字段同步
Apple -> 服务端本轮只允许最小回写：
- completed -> done

不支持：
- Apple 改标题回写
- Apple 改 bucket / category / note 回写
- Apple 删除直接回写服务端删除

### 不做复杂冲突协商 UI
本轮冲突策略以系统规则自动裁决为主，不做用户可视化冲突中心。

## 七、核心业务语义

## 7.1 任务状态

### open
- 当前未完成
- 进入主清单
- 参与同步

### done
- 已完成
- 不进入主清单
- 进入完成区 / 回顾区
- 可同步为 Apple completed

### deleted
- 真删除
- 不进入主清单
- 不进入 done / review 常规视图
- 默认不立即要求 Apple 硬删除

结论：

> `done != deleted`

---

## 7.2 分类体系
沿用现有 5 类：
- inbox
- project
- next_action
- waiting_for
- maybe

bucket 继续只保留时间语义：
- today
- tomorrow
- future
- archive

---

## 7.3 时间语义
GTD 时间解释继续统一采用：
- `Asia/Shanghai`（UTC+8，北京时间）

适用于：
- 今天 / 明天 / 下周
- 任务归类
- 视图渲染
- 提醒语义

---

## 八、同步模型要求

## 8.1 服务端 -> Mac
Mac 需要支持：
- 按 `since` 获取增量任务变更
- 获取新增 / 更新 / 完成 / 删除标记
- 基于服务端版本进行同步

## 8.2 Mac -> Apple
Mac 需要支持：
- create reminder
- update reminder
- move reminder list
- complete reminder
- 可选 soft delete / archive

## 8.3 Apple -> 服务端
本轮只允许：
- Apple completed -> 服务端 done

不允许：
- Apple 任意字段覆盖服务端
- Apple 删除覆盖服务端删除

---

## 九、数据与兼容要求

### 9.1 复用现有任务字段
应尽量保留现有字段：
- id
- title
- status
- bucket
- quadrant
- tags
- note
- category
- created_at
- updated_at
- completed_at
- deleted_at
- source
- source_task_id
- sync_version
- last_synced_at

### 9.2 保持本地兼容层
本地文件可保留为以下用途：
- 本地渲染缓存
- 离线查看
- 调试/备份导出

但不再承担主主库责任。

### 9.3 支持增量同步状态
系统需要维护：
- 任务版本号或更新时间戳
- Mac 同步游标
- Apple Reminder 映射关系
- 最后同步方向 / 状态

## 十、验收标准

满足以下条件，视为本轮版本可验收：

### 验收 1：AI 写入闭环成立
用户说一句自然语言任务后：
- 服务端成功新增任务
- 能立即查到
- 用户无需手工 GitHub 提交

### 验收 2：任务修改闭环成立
用户修改任务后：
- 服务端状态正确更新
- Mac 在同步周期内拿到变更
- Apple Reminders 正确反映更新

### 验收 3：完成闭环成立
- 在 AI 侧完成任务，Apple 端能同步完成
- 在 Apple 侧完成任务，服务端能回写 done

### 验收 4：本地视图仍可用
- `today.md`
- `inbox.md`
- `done.md`
- `weekly/review-latest.md`

均可由服务端数据正确生成

### 验收 5：GitHub 不再处于主链路
- 日常任务使用不依赖 GitHub push/pull
- 即使 GitHub 不可用，主业务流程仍可运行

---

## 十一、版本边界总结

本轮版本不是在做：
- 通用 SaaS 任务平台
- 完整双向同步引擎
- 多端协同生态

本轮版本真正要做的是：

> **把现有 GTD 系统从“文件同步思路”升级为“服务端主库思路”。**

一句话总结：

> 用户只管说任务，系统负责入库、同步和落地，GitHub 不再是日常使用的必经步骤。
